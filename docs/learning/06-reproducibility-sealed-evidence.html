Reproducibility And Sealed Evidence

What must travel with a reported number so another person can check it without trusting us?

Beginner levelTamper-check laboratoryArtifact verification
The opening question

Is "$100 became $110" a result or only a sentence?

A researcher says a strategy turned $100 into $110. You ask for the daily returns, the positions, the cash series, transaction costs, model settings, code version, and calculation. The researcher has only a screenshot. The number may be honest, but you cannot determine which inputs produced it, whether the calculation can be repeated, or whether the screenshot was changed after the run.

Unsealed claim

"Ending wealth: $110"

No return path, config, code identity, or audit record accompanies the number. Recalculation is impossible.

Checkable evidence

$100 × 1.05 × 0.98 × 1.068

The three returns, exact formula, config, source hashes, and code revision are preserved. Another person can recompute the ending wealth.

The second package is not automatically scientifically valid. The data could measure the wrong instrument, the strategy could look into the future, or the hypothesis could be poorly designed. Sealing answers a narrower but essential question: are these exactly the inputs, code, rules, and outputs that belong to this run, and do the stored summaries follow from the detailed evidence?

The idea to rememberA result is not reproducible because a command exists. Reproducibility requires frozen meaning, identifiable inputs and code, complete artifacts, and an independent path that recalculates important claims instead of trusting stored summaries.
Words before notation

From a research plan to an evidence package

Frozen contract

A specification fixed before result-bearing output is viewed. It names data, dates, models, grids, timing, costs, metrics, gates, and stopping rules.

Configuration lock

The exact machine-readable settings copied into a run. It prevents a later config edit from silently changing what the old run means.

Data manifest

A structured record of source files, identifiers, hashes, coverage, quality facts, and retrieval metadata. It points to the exact input snapshot.

Git SHA

The identifier of one committed source-tree state. It says which tracked implementation produced the run, assuming the tree was clean.

Artifact

A file produced by a run: states, validation surfaces, selections, trade paths, metrics, claim decisions, metadata, or reports.

Inventory

A complete path-to-hash map for immutable run files. Missing, extra, or modified files cause verification to fail.

Computational reproducibility means rerunning or recomputing from an immutable input snapshot and software identity. Source reproducibility means repeating the provider query; it may return revised bytes later. Conceptual replication means retesting the method when the original snapshot cannot be obtained. Chapter 5 showed why these levels must not be given the same label.

A hash is a fixed-length fingerprint computed from bytes. The project uses SHA-256. If one byte changes, the fingerprint is expected to change dramatically. A hash is excellent for detecting accidental or unauthorized modification. It does not explain what the bytes mean, certify that the original file was correct, or prove who created it.

Immutable means a completed run is not edited in place. A changed config or source snapshot receives a new identity. Deterministic means the same declared inputs and procedure produce the same output bytes or numerically specified result. Fail closed means uncertainty or mismatch stops verification instead of quietly accepting the closest available file.

An independent verifier reads the saved evidence as a skeptical reviewer. It should not merely print the stored metric table. It checks identity locks, file inventory, schema, date coverage, accounting identities, signal delay, common samples, recomputed metrics, and the claim rule. Independence here means a separate checking path inside the software, not a claim that another institution has audited the project.

Rerunning and verifying are different operations. A rerun executes models again and may expose hardware, library, or optimizer variation. Verification asks whether one already completed package is internally consistent. A strong project supports both, but verification should not require repeating an expensive fit merely to check simple accounting. Detailed artifacts let a reviewer recalculate downstream evidence cheaply while preserving the exact model output that was originally used.

A verification receipt is a compact statement produced after checks pass: run identity, terminal status, inventory size, completed coverage, recomputation difference, and conclusion. It is useful for communication, but it is not the evidence itself. Anyone evaluating the receipt must be able to point back to the immutable files and the verifier version that produced it.

Consider a three-day toy path. Day returns are +5%, -2%, and +6.8%. Starting from $100, the transparent calculation is $100 multiplied by 1.05, then 0.98, then 1.068. Anyone can recompute the endpoint. To reproduce a strategy result, however, we also need to know which position generated each return, whether cash replaced the market, whether a switch cost was subtracted, and when each signal became actionable.

The detailed trade table is therefore more valuable than a final equity chart. It preserves date, risky return, cash return, signal, delayed position, gross return, turnover, cost, and net strategy return. The metric table is a compact summary derived from this path. If a stored Sharpe ratio changes while the trade path does not, independent recomputation should reject the package.

Lifecycle state also matters. A running package may contain valid checkpoints but does not yet support a result. A boundary-failed package stopped at a preregistered search-grid gate before metrics were opened; it supports that diagnostic only. A complete package contains every required market, model, delay, path, metric, and claim file. An interrupted directory must never be promoted to complete simply because some attractive outputs already exist.

Checkpoints are mutable while a run is active because new progress must be saved. Their identity metadata and payload hashes still prevent incompatible reuse. Once the run reaches a terminal state, the final evidence is inventoried and treated as immutable. This distinction permits practical recovery from interruption without allowing a completed result to drift.

A common failure: replacing the seal after editing the evidence

Suppose someone changes a metric CSV and then regenerates the inventory. The new file hash now matches the new inventory, so a hash-only check passes. But the metric no longer agrees with the unchanged trade path. A real verifier recalculates metrics from trades and catches the inconsistency. This is why the project combines file identity with semantic and arithmetic checks.

The reverse failure also matters. If the correct trade file is copied into an old run directory without updating the inventory, the arithmetic might be valid but the package is still rejected. The verifier cannot silently infer whether the copy was intentional, whether another file is missing, or whether identities were mixed. Create a new run identity rather than repairing completed evidence by hand.

Connection to the named paper

The paper defines the study; sealing preserves one implementation

The exact source remains Shu, Yu, and Mulvey (2024), arXiv:2402.05272v3. Section 2 defines the broad data objects and dates. Sections 3.3 and 3.4 define HMM and Jump Model procedures. Section 3.4.3 describes monthly time-series cross-validation over an eight-year lookback, followed by out-of-sample use. Section 4 and Tables 4-5 present performance and delay evidence.

Those descriptions are the scientific source of truth for the protocol comparison. A versioned PDF hash ensures we do not accidentally quote a different revision. But the paper itself is not this project's artifact format, and this chapter does not claim the authors used our run ID, inventory, or verifier. Those are engineering controls added so our interpretation can be audited precisely.

A paper can be scientifically informative without publishing every raw byte, especially when data licenses restrict redistribution. Reproduction then requires documented acquisition procedures, exact identifiers where possible, and author communication for unresolved details. Sealing cannot manufacture inaccessible Bloomberg or GFD data. It preserves the proxy inputs we actually used and prevents us from later describing them as something else.

Connection to this project

Three identities create the run name

The active runner in src/adaptive_jump/cli.py combines the frozen config hash, exact data-manifest hash, and Git SHA. The first twelve hexadecimal characters of each become part of the run directory name. The run also copies config.lock.toml and data-manifest.json, records package versions and lifecycle status, and rejects an existing directory whose locks disagree.

identity = {
    "config_sha256": config.sha256,
    "data_manifest_sha256": frozen.sha256,
    "git_sha": git_sha,
}
run_id = "fixed-baselines-" + "-".join(value[:12] for value in identity.values())

Checkpoint reuse follows the same identity. An interrupted expensive HMM run can resume only when payload hash, config hash, data hash, and Git SHA match. A completed or boundary-failed run is immutable. Generated reports live outside the run because they can be regenerated; they are presentation, not the primary evidence.

The verifier in src/adaptive_jump/artifacts.py first recomputes the inventory, then checks identity locks and boundary coverage. For a complete run it reads every expected trade path, confirms binary signals and positions, recalculates gross return, turnover, cost, delayed alignment, and performance metrics, and rebuilds the directional claim. The CLI workflow is documented in the README.

Tests deliberately tamper with canonical data, metric files, and inventories. They require the loader or verifier to reject the package. Another test modifies a metric, regenerates the inventory, and still expects metric recomputation to find the inconsistency. These tests validate the checking machinery; they do not validate the paper's hypothesis.

Mathematical layer

Identity first, then recomputation

Let b be the ordered bytes of a file and H the SHA-256 function. Its fingerprint is:

An inventory is a mapping from each relative path pj to an expected hash hj. Inventory verification requires both the path set and every fingerprint to match. The logical operation is a conjunction: all files must pass. One missing file, one extra file, or one changed fingerprint makes the package fail.

V=I(Pactual=Pexpected)×∏j=1nI(H(bj)=hj)

Finally, let detailed trade evidence be T, frozen settings be C, the metric function be F, and the stored metric be mstored. Verification recomputes mnew=F(T,C) and requires the absolute difference to remain within a frozen numerical tolerance. This checks derivation, not merely file identity.

SHA-256 collisions are theoretically possible but are not the practical risk addressed here. The ordinary research risks are wrong files, accidental edits, mixed run directories, stale checkpoints, changed configs, and summaries that no longer follow from detailed evidence. More importantly, a perfect hash match says nothing about causal correctness or data semantics. Those need the protocol checks from Chapters 5 and 7.

Evidence and limitations

A seal protects identity, not truth

What sealing establishes

The exact config, manifest, code identity, file set, detailed paths, stored metrics, and claim decision agree under the implemented verifier.

What sealing cannot establish

That proxy data equal the paper data, that a model captures a real economic state, that the holdout remains untouched, or that a claim is universally true.

Current project evidenceThe frozen v7 run has a verifier-readable artifact package and a deterministic English report generated only after verification. Current values should be read from that generated report through the README workflow, not copied into this lesson.
Remaining boundaryComputational integrity makes a negative or positive result auditable. It does not upgrade a proxy replication to exact reproduction and does not unblock adaptive research after a failed scientific gate.

Long-term reproduction also depends on environment and access. The project pins Python packages in uv.lock, but operating systems, CPU libraries, provider APIs, and licensed files can still change. Preserving source snapshots and package metadata reduces uncertainty without promising that every future machine will reproduce every floating-point bit forever.

A narrative report and an experiment registry serve different audit roles. The report explains one verified run to a human and can be regenerated. The registry records that the experiment existed, what sample it accessed, and whether later work was result-driven. Neither substitutes for trade-level artifacts. Conversely, a directory full of CSV files without a frozen question and registry history cannot show whether the same holdout was inspected repeatedly while the model changed.

Calculation: transparent ending wealth

Start with $100 and apply +5%, -2%, and +6.8%. Wealth is $100 × 1.05 × 0.98 × 1.068 = $109.89 after rounding to cents. Preserve full precision in the artifact and round only for display.

Error diagnosis: "the command ran, so the study is reproducible"

A command can run against different data, defaults, code, or package versions. Reproducibility requires resolved inputs, a frozen contract, identity records, complete outputs, and verification of the real path.

Cumulative recall from Chapters 1-5

The sealed trade path must preserve market-or-cash positions, total and cash returns, costs, wealth inputs, and data definitions. A hash can protect a price-index proxy exactly while Chapter 5 still tells us it is not the paper's total-return series.

Advisor explanation

"Our run name binds config, data manifest, and Git revision. The inventory detects file changes, while the verifier recomputes accounting, metrics, and the claim from detailed paths. This establishes internal computational integrity, not exact data parity or scientific truth."

How to explain this to your advisor

Short version: "I treat a reported metric as the end of an evidence chain. The frozen contract, exact data manifest, code revision, detailed trades, inventory, and independent recomputation must agree."

Important qualification: "A sealed error is still an error. Integrity controls let us detect drift and audit calculations; semantic validity and causal research design require separate evidence."
